#Project Overview
This project simulates an operational rule-based transaction monitoring engine used in a Canadian retail banking environment to identify potential money laundering, fraud, and terrorist financing typologies. Using synthetic transactional data, I built a PostgreSQL engine to automate risk-based alert generation and exported these alerts to Microsoft Excel to construct an interactive compliance triage dashboard.

The objective is to establish a framework for identifying suspicious activities under Canada's Proceeds of Crime (Money Laundering) and Terrorist Financing Act (PCMLTFA) and FINTRAC expectations.

#Technology Stack
Database Engine: PostgreSQL (Common Table Expressions, Window Functions, Self-Joins)

Analysis & Visualization: Microsoft Excel (VLOOKUP, Pivot Tables, Conditional Formatting, Slicers)

Methodology: Financial Intelligence Unit (FIU) alert triage and transaction velocity checks

#AML Detection Scenarios Implemented (SQL)
1. Structuring Detection
Identifies transaction patterns where cash deposits are structured to remain just below the $10,000 FINTRAC regulatory reporting threshold.

SQL Logic: Rolling window summation checks for cash transactions totaling $9,000 to $9,999.

2. Rapid Layering & Pass-Through Activity
Detects high-velocity, circular money movement indicating layering.

SQL Logic: Identifies accounts where a deposit from Party A is immediately forwarded (98%+ of the value) to Party C.

3. High-Risk Jurisdiction Outbound Wires
Flags international wires routed to high-risk or FATF-sanctioned jurisdictions.

#Excel Compliance Triage Dashboard
The alerts generated by the PostgreSQL engine were imported into Excel for risk triaging:

Data Enrichment: Used VLOOKUP to integrate customer database metrics (PEP status, geographical risk) with transactional alerts.

Interactive Triage: Built Pivot Tables and Slicers allowing compliance officers to filter alerts by risk tier (High, Medium, Low) and branch.

Risk Heat Mapping: Applied conditional formatting to instantly surface "critical exceptions" requiring immediate escalation or Suspicious Transaction Report (STR) drafting.

#Key Insights & Quantified Results
Processed a mock dataset of 6,000,000 transactions.

The SQL engine successfully generated 7984 alerts, filtering out 99.7% of "noise" (false positives).

Identified 26 high-risk structuring events for mock STR filing.
